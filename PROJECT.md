#  Контекст проекта

Проект — local-first PWA-приложение для ведения личной библиотеки фильмов и сериалов. Пользователь добавляет тайтлы вручную, указывает тип, статус просмотра, рейтинг, месяц и год просмотра.

Главный фокус проекта: скорость работы, сохранность пользовательских данных, простая архитектура, понятная синхронизация и инкрементальное развитие.

## Философия продукта

Приложение работает от локального состояния.

UI читает данные из IndexedDB и записывает пользовательские изменения в IndexedDB.

Сервер отвечает за авторизацию, облачную копию данных и синхронизацию.

Пользовательские изменения сначала сохраняются локально. После этого они синхронизируются с сервером.

Приложение сохраняет рабочее состояние в offline-режиме после успешной авторизации и загрузки данных.

Архитектура развивается постепенно. Каждый этап должен давать работающее приложение с понятной ценностью.

## Текущий MVP

В первой версии у пользователя есть один личный список фильмов и сериалов.

Пользователь авторизуется через Telegram OIDC.

Тайтлы вводятся вручную.

Для тайтла нужны поля:

- `itemId`
- `kind`: `movie` / `series`
- `title`
- `status`
- `rating`
- `watchedYear`
- `watchedMonth`
- `posterUrl`

Коллекции присутствуют в модели с первого этапа. В UI на старте используется одна коллекция пользователя.

`library_items` принадлежат коллекции через `collectionId`.

## Будущее развитие

Модель коллекций должна поддерживать расширение до shared collections.

В будущем у пользователя может быть личная коллекция и совместные коллекции, например список “смотрим вместе”.

Расширение до совместных коллекций должно происходить через добавление участников коллекции, ролей и управления доступом.

## Архитектурные правила

UI работает от локальной базы.

Каждое пользовательское изменение проходит через mutation layer.

Mutation layer в одной локальной транзакции обновляет локальные данные и добавляет операцию в outbox.

Outbox хранит операции, которые ожидают синхронизации с сервером.

Sync client отправляет pending operations на сервер.

Сервер применяет операции, сохраняет результат и возвращает актуальный snapshot коллекции.

Клиент применяет server snapshot к локальной базе с сохранением локальных pending changes.

Синхронизация использует модель:

```text
client → server: pending operations
server → client: collection snapshot
````

Эта модель называется:

```text
operations up, snapshot down
```

## Sync MVP

Минимальные операции:

* `item.upsert`
* `item.delete`

`item.upsert` передает актуальное состояние item целиком.

`item.delete` фиксирует удаление item.

Клиент генерирует постоянные id:

* `itemId` для тайтла
* `opId` для операции
* `collectionId` для коллекции

`opId` используется для безопасных повторов.

Сервер хранит обработанные операции в `processed_operations`.

При повторной отправке операции с тем же `opId` сервер возвращает подтверждение уже обработанной операции.

Потерянный ответ сервера обрабатывается повторной отправкой операции из outbox.

## Минимальный sync contract

Client request:

```ts
type SyncRequest = {
  collectionId: string
  knownRevision: number
  operations: SyncOperation[]
}
```

Operation:

```ts
type SyncOperation =
  | {
      opId: string
      type: 'item.upsert'
      item: LibraryItemPayload
    }
  | {
      opId: string
      type: 'item.delete'
      itemId: string
    }
```

Server response:

```ts
type SyncResponse = {
  accepted: string[]
  rejected: RejectedOperation[]
  collection: CollectionSnapshot | null
}
```

`operations` — pending operations из outbox.

`collection` — актуальный snapshot коллекции.

На первой реализации сервер может возвращать collection snapshot после каждого sync.

## Серверные сущности MVP

Минимальные таблицы:

* `users`
* `sessions`
* `collections`
* `library_items`
* `processed_operations`

`users` хранит внутреннего пользователя приложения и связь с Telegram identity.

`sessions` хранит backend-сессии.

`collections` хранит списки тайтлов. На старте у пользователя одна основная коллекция.

`library_items` хранит фильмы и сериалы внутри коллекции.

`processed_operations` хранит обработанные операции для безопасных повторов sync.

## Локальные сущности MVP

В IndexedDB нужны:

* `local_collections`
* `local_library_items`
* `outbox`
* `sync_state`

`local_collections` хранит локальные коллекции.

`local_library_items` хранит локальные тайтлы.

`outbox` хранит pending operations в порядке создания.

`sync_state` хранит служебное состояние синхронизации: device id, default collection id, last sync info.

## Правила сохранности данных

Пользовательское изменение сначала фиксируется локально.

Локальная транзакция сохраняет данные и outbox operation вместе.

Sync можно безопасно повторять.

Сервер применяет каждую операцию один раз.

Клиент сохраняет pending operations до подтверждения сервером.

Server snapshot применяется аккуратно относительно локальных pending operations.

Внешний `posterUrl` считается вспомогательным отображаемым полем. Основные пользовательские данные хранятся в локальной базе и на сервере.

## Правила принятия решений

Выбирать понятную архитектуру.

Развивать проект инкрементально.

Оставлять sync-контракт маленьким и проверяемым.

Генерировать id на клиенте для local-first операций.

Использовать outbox для всех пользовательских изменений.

Использовать snapshot sync для MVP.

## Главная архитектурная формула

```text
Local-first MVP = IndexedDB + outbox + safe retry + server snapshot
```

Цель первого этапа — маленькая, надежная и понятная синхронизация, которую легко реализовать, протестировать и расширить.
